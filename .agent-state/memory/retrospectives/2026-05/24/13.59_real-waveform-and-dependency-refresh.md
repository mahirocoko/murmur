## Honest Feedback
Three friction points were clear. First, I initially implemented “real waveform” from the native samples but did not immediately consider whether the indicator window had permission to receive Tauri events. In a normal React app that omission would not exist, but in Tauri each webview has capability scope. I should have checked `capabilities/default.json` at the same time I added the new `listen()` call. Second, the first verification set was too compile-focused. `pnpm build` and `cargo check` proved the code compiled, but they did not prove the visible waveform moved in the actual indicator window. Mahiro had to supply the runtime observation. Third, dependency updating revealed a small process gap: `cargo upgrade` was not installed, and my first shell probe used a read-only zsh variable name `status`, which was noisy. I recovered by using built-in `cargo add` and `cargo update`, but I should avoid clever shell wrappers when simple commands are enough.

## Lessons Learned
- A voice indicator should not be driven only by a single RMS scalar plus synthetic animation; real microphone reactivity needs a waveform/frequency array or per-bucket analysis.
- In this Tauri app, native CPAL recording can be the source of truth for visualization data. Avoid opening a second frontend `getUserMedia` stream just to draw the indicator.
- Tauri capabilities must include every window that calls frontend APIs. If `IndicatorWindow` uses `listen()`, the `indicator` window must be in the capability scope, not only `main`.
- A passing `cargo check` or `pnpm build` is not enough for interaction-level UI behavior; custom indicator/desktop behavior needs manual runtime confirmation when possible.
- `cargo fmt --check` drift can be fixed safely with `cargo fmt`, but expect broader formatting diffs in files that were already not rustfmt-normalized.
- Updating Rust audio dependencies can introduce small API changes; after CPAL updates, verify audio config and stream code specifically.
