## AI Diary
I felt this session as a useful product-shape correction. I had implemented a separate settings window, but Mahiro's screenshot and feedback made it clear that “window exists” was not the same as “feels like Cap.” The Cap-like quality came from hierarchy: sidebar sections, a bigger settings surface, and keeping the compact main window focused on recording. I initially kept too many concepts in the compact toolbar because it was easy to reuse the old `activeView` model. Mahiro pushed the design toward a cleaner split: main is for immediate dictation, settings is for configuration/history. That forced me to unwind some earlier structure instead of layering more UI on top. I also noticed a recurring lesson with desktop apps: window visibility is part of UX logic, not a small afterthought. Hiding the main window when settings/indicator opens felt right, but it created the inverse requirement that closing settings should restore main. The final wave tweak was another good reminder that audio visualization is felt visually, not just technically correct; a too-tall idle baseline reads as noise or loading even if real data is wired correctly.

## Honest Feedback
Three friction points stood out. First, I initially approached “like Cap” too literally as a separate window plus some styling, but Cap's settings feel also comes from sidebar information architecture and from preserving the main window as a focused capture surface. I should have translated the product structure earlier, not only the window primitive. Second, the transition from `activeView` to a true multi-window model left some dead/legacy state during editing, and I had to clean up TypeScript unused variables after the build caught them. That is acceptable during refactor, but I should be more deliberate when retiring a UI state machine. Third, the show/hide behavior needed both directions. I hid main when opening settings/indicator, but Mahiro had to point out that closing settings must bring main back. For desktop UX, I should think in complete lifecycle pairs: open/hide, close/restore, tray routing, and native close event, not only the entry path.

## Lessons Learned
- “Like Cap” means matching the window hierarchy and information architecture, not just making a separate window. Settings should be a dedicated larger surface with sidebar sections.
- Main dictation window should stay focused on immediate capture: record button, status, selected output, shortcut, latest transcript, and internal mode tabs.
- History fits better in the settings/sidebar surface than as a compact toolbar tab.
- When opening a secondary window (settings or indicator) hides the main window, closing that secondary window must restore main unless the user explicitly quit/hid the app.
- Scrollbar visual weight matters in desktop custom chrome; wide default scrollbars can make a refined panel feel unfinished.
- Audio indicator idle height should be conservative. Even technically real waveforms can feel like fake loading if the baseline is too tall.
