## Honest Feedback

มี 3 friction points ที่ควรจำจริงจัง  หนึ่ง ผมเริ่มด้วย design/API shape ที่เร็วเกินไป ทำให้เกิด state แปลก เช่น catalog model กับ local model แยกกัน ทั้งที่ผู้ใช้มองว่าเป็น model เดียวกัน  ถ้าคิดจาก ownership ก่อน — Murmur-owned vs external local — น่าจะลดรอบแก้ได้  สอง ผมประเมิน Tauri command behavior ต่ำไป ตอนแรกเพิ่ม progress event แล้วคิดว่าจะพอ แต่ command ยัง sync/block อยู่ ทำให้ UX ยังเหมือนค้าง  นี่เป็น desktop-runtime gotcha ไม่ใช่แค่ React state issue  สาม เรื่อง copy/label ต้องละเอียดกว่าเว็บทั่วไป เพราะ model names, file names, source, path, size, quality/speed ทั้งหมดเป็นข้อมูลตัดสินใจของผู้ใช้ ถ้า label สวยแต่ source ไม่ชัด จะทำให้ user trust ลดลงทันที

## Lessons Learned

- Model management ต้องเริ่มจาก ownership boundary ก่อน: app-owned data, external local data, env fallback, bundle data เป็นคนละ policy
- สำหรับ Murmur ตอนนี้ model source ควรเป็น Murmur app data เท่านั้น ส่วน `whisper-cli` และ `ffmpeg` ยัง detect จากระบบได้
- Download ไฟล์ใหญ่ใน Tauri ห้ามทำเป็น sync command ที่ block UI; ต้อง async/spawn blocking และ emit progress ที่ UI render ได้จริง
- Tauri invoke arg naming จาก JS ต้องใช้ camelCase เช่น Rust `model_id` รับจาก JS `{ modelId }`
- Build/typecheck ไม่พอสำหรับ desktop UX; ต้อง manual QA ปุ่ม Download/Uninstall และ progress behavior ด้วย
- App data บน macOS ไม่หายเมื่อ drag `.app` ลง Trash ปกติ ต้องมี Uninstall/cleanup path ในแอปหรือให้ user ลบ Application Support เอง
