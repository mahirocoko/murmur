## Honest Feedback

Three friction points stood out. First, I over-reported confidence before checking all code paths. The settings window width had both config and Rust runtime overrides; fixing only one meant the user’s visual reality contradicted my summary. Next time, for Tauri window sizing I should search for every `set_size`, config width, and command path before claiming the UI size changed. Second, my first transparency attempt leaned too much on CSS `backdrop-filter`, which is a web mental model, not the full macOS desktop-window reality. Cap already had evidence that native material and transparent document roots matter; I should have consulted that earlier instead of patching opacity by feel. Third, I let a bit of UI ceremony survive too long: mode tabs, debug notices, and one-item nav abstractions made the app feel heavier than its purpose. Mahiro had to steer it toward minimalism. The course correction was good, but the lesson is that Murmur’s main surface should be treated as a capture appliance, not a dashboard.

## Lessons Learned

- For Tauri windows, check both static config and Rust/JS runtime resizing paths before reporting size changes.
- macOS translucency is native-material-first: `transparent: true`, transparent document/body/root, native `setEffects`/`windowEffects`, then low-opacity CSS tint layers. CSS `backdrop-filter` alone will not prove desktop blur.
- Murmur’s main window should stay minimal and action-first. Settings/history/model management belong in the dedicated settings window.
- Direct CLI review lanes are useful after a first local pass, especially for catching dead UI state and semantic/a11y seams.
- UI polish should remove debug/filler UI before adding visual detail.
