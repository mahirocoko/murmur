## Blockers & Resolutions

- **Clustered bars**: Caused by repeated `LOADING_WAVE_PATTERN`; resolved with a continuous bell/envelope across normalized bar position.
- **Bulky/too fat wave**: Caused by a broad envelope plus shoulder; resolved by narrowing the envelope, reducing shoulder, lowering idle, and thinning bars.
- **State transition restart**: Likely caused by state-specific animation duration; resolved by using a fixed duration across loading states.

## Honest Feedback

Three friction points are worth keeping. First, I was still treating the waveform as an implementation exercise instead of a perceptual object. The repeated pattern was easy to code and easy to reason about, but it did not match the reference's continuous body. Second, build verification is not enough for animation quality. It is helpful, but I should keep saying “build-passed, not visually runtime-confirmed” unless I actually run and observe the indicator. Third, I need to preserve continuity as a first-class requirement for stateful UI. A small prop or CSS variable difference such as duration can create a perceptible reset even if the visual shape looks correct. The user should not have to point out discontinuity every time; I should inspect state-to-state animation invariants myself.

## Lessons Learned

- For waveform-like loading UI, a repeated peak pattern can look like disconnected clusters. Use a continuous envelope for a single connected shape.
- If a state transition should feel continuous, keep animation name, duration, delay strategy, and keyframes stable across states.
- Make visual tuning changes by adjusting a few named levers: envelope center/width, shoulder, idle scale, bar width, and duration.
- Build checks do not prove animation feel; label that limitation clearly.
