## Architecture Decisions

- Shortcut customization is a real native preference, not just a UI label. The Rust side owns OS registration so shortcut behavior matches what the UI displays.
- Shortcut capture temporarily unregisters the currently active dictation shortcut because an already-registered global shortcut can prevent the WebView from seeing the key combo.
- Permission toggles are deliberately “switch-style setup rows,” not false OS toggles. macOS still requires the user to grant permissions in System Settings.
- Accessibility can be checked accurately; Microphone is approximated from available input devices for now, and Global Shortcut is treated as active while the app is running.
- The OG image source of truth for the landing site is `site/public/og-image.png`; root `public/og-image.png` exists but is not the primary site metadata asset.

## What Could Improve

- I should have predicted the global shortcut capture trap earlier. In desktop apps, a global hotkey being active while trying to record a new hotkey is a classic footgun.
- Some Agy-generated CSS was too broad and comment-heavy; I cleaned part of it, but future passes should be tighter from the start.
- Permission status is still mixed: Accessibility is true native status, while Microphone and Global Shortcut are inferred/operational. The UI should eventually distinguish “verified by OS” vs “ready in app” more explicitly.

## Blockers & Resolutions

- **Shortcut could not be reset while in capture mode**: resolved by adding native unregister/restore commands and window-level key handling.
- **Accessibility switch was fake/off-only**: resolved with `AXIsProcessTrusted()` native check.
- **Selected rail was visually heavy**: resolved by removing thick border/rail and using subtler selected background + check icon.

## Honest Feedback

There were three friction points. First, the direct CLI lane sometimes produced implementation plans/artifacts and over-reported “staged” work even though nothing was staged; I had to treat pane output as useful but not fully authoritative. Second, I accepted a broad implementation pass from Agy and then needed to inspect it carefully because some changes were “technically working” but product-wrong, such as selectable Soon modes and heavy selected borders. Third, I initially under-tested the shortcut customization as a real desktop interaction. Build checks and TypeScript were green, but Mahiro’s screenshot and feedback showed the UI was still stuck in `Press shortcut…`. The durable lesson is that desktop interaction features need runtime mental-model testing: focus, blur, global OS registration, permission prompts, and event propagation all matter. I should proactively test these flows or design around their failure modes before claiming they work.

## Lessons Learned

- Custom global shortcut capture must pause the active registered shortcut before listening for a replacement.
- macOS Accessibility permission should be queried with native APIs instead of represented as a static setup row.
- Selected states should be explicit but calm; check icon + subtle background can be clearer than a thick accent rail in Murmur’s compact translucent UI.
- Home should expose the primary action shortcut because dictation apps are shortcut-first products.
- OG image wiring for the landing site belongs in `site/public` + Astro metadata; README preview is a nice documentation touch, not app runtime config.
