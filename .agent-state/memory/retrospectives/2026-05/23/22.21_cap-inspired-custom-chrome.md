## Honest Feedback
Three friction points stood out. First, I moved too fast from visual imitation to implementation without identifying all Tauri runtime constraints. I knew `data-tauri-drag-region` existed but forgot that Tauri v2 capabilities can silently block the underlying command. That led to several noisy attempts with CSS `-webkit-app-region`, manual `startDragging()`, and no-drag toggles before finding the real missing piece. Second, I let Gemini contribute useful edits but did not constrain its state enough; it introduced a duplicate tail and stale assumptions, so I had to clean up after it. Direct CLI can help, but only when I keep it narrow and continuously verify pane output against current files. Third, I reported some fixes as verified after build/cargo checks even when the bug was interaction-level. Build success proved code compiled, not that close/drag worked. For custom desktop chrome, manual/runtime behavior must be treated as first-class verification, and when I cannot personally click the app, I should phrase confidence more carefully.

## Lessons Learned
- Cap’s desktop toolbar is custom HTML, not native, but it works because the Tauri window setup and drag-region permissions are correct.
- `data-tauri-drag-region` in Tauri v2 needs capability `core:window:allow-start-dragging`; otherwise the DOM can look correct but dragging silently fails.
- Transparent rounded macOS Tauri windows need a bundle of settings: `transparent`, macOS private API/feature, window effects/material, transparent background color, and inner shell radius/overflow.
- For custom close buttons, stop propagation and keep the button out of the drag region. A dedicated backend hide command can be more reliable than relying on close event behavior.
- Global shortcuts should be registered safely. Dynamic register/unregister during live recording can create fragility; pre-registering Esc and only acting when recording is safer.
- Cap must be the first reference for future Mahiro Whisper custom chrome/UI behavior.
