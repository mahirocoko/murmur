## Architecture Decisions

- Keep Tauri for now because it is fast to iterate product/UX and enough for personal use.
- Move system-critical behavior out of React/webview as soon as the workflow requires background reliability.
- Treat the main window as Control Center only; tray + shortcut are the primary interaction surface.
- Prefer multilingual models for Thai/mixed-language dictation; avoid `.en.bin` unless the user intentionally chooses English-only.
- Keep native Swift/AppKit as a possible future direction if latency, Accessibility, or background behavior becomes too constrained.

## Honest Feedback

First friction point: I let “build passes” stand in for “workflow works” too often. A dictation app is judged by whether it preserves focus, indicates state clearly, and inserts text where the user expects. Passing `cargo check` is necessary but not enough. Second friction point: I introduced UI states before guaranteeing the underlying event path was reliable. That caused the indicator to stay stuck on Recording, which is exactly the kind of product polish bug Mahiro notices immediately. Third friction point: I leaned on generic Tauri patterns before fully respecting macOS-native behavior. Tray menus, Accessibility, CoreGraphics, clipboard encoding, and background audio all have native constraints. The solution was not “more React UI”; it was moving the correct responsibilities to native code and making the webview a control surface. I should carry that lesson forward and ask “what owns this behavior when the app window is closed?” before choosing implementation boundaries.

## Lessons Learned

- For menu bar dictation apps, the main window must not own the core recording lifecycle.
- Clipboard and paste are separate concerns: Unicode-safe copying can succeed while paste injection still fails due to Accessibility.
- Whisper language selection and model selection are distinct product controls; mixed Thai+English needs a multilingual model plus language/prompt bias.
- Floating indicators need state replay/delayed emits because secondary webviews may not be ready when the first event fires.
- “Seamless” means preserving focus and giving feedback during every slow phase: listening, transcribing, pasting, done/error.
