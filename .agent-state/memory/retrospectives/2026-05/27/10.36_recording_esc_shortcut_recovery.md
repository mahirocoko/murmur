## Session Summary

This session started from a product question: why did Murmur remove `Esc` from the indicator flow? I explained the previous reason—indicator is non-focusable and global `Esc` had been removed because it could consume `Esc` in other apps. Mahiro then asked to bring it back only during recording, which exposed the old gotcha again: a synchronous global shortcut registration inside the `Option + Space` shortcut path caused recording to hang. I corrected the lifecycle so `Esc` registration/unregistration runs off the shortcut handler path, updated docs, added the `Esc` hint to the indicator UI, and validated with Rust/TypeScript checks.

## AI Diary

This session was a good reminder that a remembered gotcha is only useful if I design around the exact failure mode, not just mention it in prose. I knew from memory and docs that a previous global `Esc` attempt had made recording hang. When Mahiro asked to bring `Esc` back only during recording, I chose the right product behavior but initially implemented it too directly: register `Esc` right after starting the stream and unregister it on stop/cancel. It compiled, but Mahiro immediately found the real-world failure by pressing the shortcut. The important part was not to defend the implementation because it passed checks. The symptom matched the old footgun: shortcut plugin methods schedule work on the main thread, and doing that synchronously while inside another shortcut callback can deadlock or feel frozen. I corrected the model: the `Option + Space` handler must return quickly, and `Esc` registration can happen just after, on another thread, with a recording-state guard. I also liked that Mahiro clarified the UI request precisely: add `Esc` to the indicator UI only, not more native behavior. That helped keep the UI change small. I should keep treating manual shortcut/window QA as required for Tauri behavior; compile checks are not enough for global shortcut lifecycle.

## Honest Feedback

1. My first recording-only `Esc` implementation repeated the class of mistake already recorded in memory. I remembered the risk but still used synchronous registration in the start/stop path. The better default should have been “never block inside global shortcut handler paths” from the start.
2. I initially reported the feature as done after build/lint/cargo checks, but shortcut behavior must be manually tested in `pnpm tauri dev`. The freeze proves why these checks are only compile confidence, not interaction confidence.
3. The docs update was directionally right but temporarily got ahead of verified behavior. When changing shortcut behavior, docs should follow after the native lifecycle has survived at least the basic record/cancel/stop path.
4. The indicator UI hint is intentionally small and clear, but it still needs visual QA in the actual pill because footer space is limited and `Esc cancel | ⌥ Space stop` could crowd on small indicator widths.

## Lessons Learned

- Do not synchronously call Tauri global shortcut `on_shortcut`/`unregister` from inside another global shortcut handler path. Spawn or defer the work so the original handler can return.
- Recording-only global shortcuts need state checks both before and after registration because recording can stop/cancel while the registration task is pending.
- For Tauri shortcut/window behavior, `cargo check` and `pnpm build` are not enough. Manual QA is part of correctness.
- UI-only indicator affordances should stay UI-only unless Mahiro explicitly asks to change native behavior.
- Keep docs honest: if behavior has a manual QA risk, report it as such rather than implying compile checks proved the flow.
