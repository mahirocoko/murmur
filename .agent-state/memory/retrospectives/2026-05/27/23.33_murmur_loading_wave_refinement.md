## AI Diary

I felt the familiar discomfort of visual work where “technically plausible” is not the same as “visually right.” At the beginning I over-indexed on making the wave smoother by reasoning from code instead of holding the reference video and Mahiro's perception as the actual target. The first few passes were too abstract: I used wave math, delays, and phase language, but the user still experienced it as not right. That is an important signal. The session improved when I stopped pretending certainty and admitted that I had only inspected extracted frames rather than truly matching a moving reference. Cursor Opus helped by grounding the visual target more precisely: the Cap reference had thin dense bars with traveling swell envelopes, not merely bars translating up and down. Even then, the “technically smoother” rAF approach created a visual that Mahiro found stranger, which reminded me that performance optimization can change the feel. The better recovery was to simplify the perceptual mechanism: keep a fixed bar height and animate transform scale instead of reflowing heights. This session was productive, but also humbling. I should ask for visual confirmation sooner and report uncertainty more plainly when I am approximating motion from code and frames.

## What Could Improve

- The Gemini lane produced a partial CSS corruption (`se, -3px))...`) that I caught, but I should have contained external edits more cautiously.

## Blockers & Resolutions

- **Reference ambiguity**: The video motion was hard to describe from a short clip. Resolved by extracting multiple frames and later letting Opus inspect the video directly.
- **Visual mismatch**: Several mathematically reasonable waveforms still did not match Mahiro's taste. Resolved through iterative feedback rather than one-shot design.
- **Motion jitter/strangeness**: Height writes every frame felt odd. Resolved by switching to `transform: scaleY(...)` and reducing density.

## Honest Feedback

Three friction points stand out. First, I need to stop turning UI bugs into style doctrine unless the user asks; Mahiro immediately corrected me with “ใจเย็น ตอนนี้มันบัคนะ,” and that was fair. Second, I overclaimed “ตาม video” before I had enough evidence. For motion work, static frames plus code intuition are not enough; I should say “approximation” until I have runtime confirmation. Third, delegation needs tighter review. Gemini and Opus were useful, but both produced changes that needed strong human/agent judgment afterward. Gemini caused a syntax break; Opus improved the reference model but later introduced a visual approach that felt worse. The correct role for sub/CLI agents here is not “they solved it,” but “they supplied evidence and alternatives.” I need to synthesize more skeptically, especially for UI motion where small timing and density choices matter more than architectural elegance.

## Lessons Learned

- For visual motion bugs, reference video evidence must guide the implementation more than verbal labels like “smooth” or “spiral.”
- “Smooth” does not always mean more direct per-frame DOM updates; changing layout properties every frame can feel worse than transform-based animation.
- If the user says the result is only partly improved, preserve that as current reality and stop before over-tuning if they choose to stop.
- Direct CLI agents are useful for fresh eyes but still require local verification and taste review.
