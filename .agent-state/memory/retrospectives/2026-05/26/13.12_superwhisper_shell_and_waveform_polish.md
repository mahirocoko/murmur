## Honest Feedback

Three friction points mattered. First, using Cursor Opus for a large UI pass produced a lot of code movement at once. It was efficient, but it also increased the burden on me to inspect for reference leakage and stale remnants. I caught and removed the settings window later, but ideally I should have declared the shell architecture decision earlier: either integrated settings or separate settings, not both. Second, the sidebar collapse bug was predictable: when narrowing a sidebar to icon-only, non-nav decorations such as traffic lights, brand marks, and footer state also need collapsed-state rules. I added the nav behavior first but missed the overflow until Mahiro showed the screenshot. Third, waveform behavior required multiple rounds because I initially interpreted “flow” as decay plus scrolling. Mahiro wanted recorded bars to remain as historical speech marks until they leave the viewport. That is a subtler state model, and I should have asked myself whether the visualization represented current loudness or a timeline of captured audio before implementing.

## Lessons Learned

- When adopting a reference UI, copy anatomy and information architecture, not brand or fake product data.
- If main shell contains settings sections, remove the standalone settings window promptly to avoid duplicate routes and stale lifecycle code.
- Collapsed sidebar QA must include every sidebar child: traffic controls, brand, nav labels, footer/status, separators, and hit targets.
- Tauri macOS material can change on focus if `followsWindowActiveState` is used; use `active` when the product wants consistent focus/unfocus opacity.
- Waveform UI has two different models: instantaneous equalizer vs rolling recording timeline. Mahiro prefers the recording timeline for Murmur's indicator.
- For rolling waveform, append new bars and slide older bars rather than replacing the whole row; do not decay history unless the user wants fading.
