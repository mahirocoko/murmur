## Honest Feedback

Three friction points deserve remembering. First, the initial site file write failed because `site/src/pages` did not exist. It was a small operational miss, but it is exactly the kind of thing that wastes momentum during setup. I should create the full directory tree before writing multi-file project scaffolds. Second, the first Tailwind integration choice was too optimistic. `@tailwindcss/vite` is the normal modern path, but with current Vite 8 it failed during build. I recovered quickly with `@tailwindcss/postcss`, yet I should describe this as a compatibility-driven choice rather than pretending the original plan was perfect. Third, I did not immediately enforce exact versions in the root package, even though Mahiro likes fixed versions in several repos. Mahiro explicitly corrected it, and I fixed it. This should become a durable local convention for Murmur/site work: use exact package versions and do not add `^` unless Mahiro asks.

## Lessons Learned

- For Murmur, repo/package dependencies should be pinned to exact versions; latest means “install the current latest exact version,” not caret ranges.
- Astro + Tailwind 4 + latest Vite may need PostCSS integration if `@tailwindcss/vite` hits build-time binding issues.
- A desktop app landing page should be separated under `site/` so Tauri/Vite app code stays focused.
- shadcn-style in Astro can be implemented as small primitives (`Button`, `Card`, `Badge`) with CVA + `cn()` without pulling in a full React component stack.
- The Murmur landing page works best when it uses product-specific truth: local-first, whisper.cpp, Thai+English workflow, shortcut, model library, privacy, build-from-source.
- Good reference/taste work comes from restraint: no fake SaaS metrics, no decorative glows, no exaggerated claims.
